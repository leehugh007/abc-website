## 1. 漏斗總圖

每個工具的標準漏斗有 4 階段（除非該工具沒 claim 流程）：

```
看結果（complete）
  ↓ 願意填進階？
進階（start_advanced）
  ↓ 完成進階？
領代碼（claim_generated）— 寫 Supabase + 顯示代碼
  ↓ 點 LINE？
點 LINE（click_line_<tool>）— 帶代碼到 LINE 領完整報告
```

> **fatty-liver 例外（2026-04-29 起）**：沒進階，看結果直接 await `generateClaim` → claim_generated 跟 complete 幾乎同時發生

每個工具的階段覆蓋見 §3 事件矩陣。

## 2. 工具清單與設計差異

| 路由 | 工具 | 表單型態 | 進階？ | claim_code 流程？ | 主推路徑 |
|------|------|---|:-:|:-:|---|
| `/tools` | TDEE / BMR | 計算型（gender + 年齡身高體重 + 活動量） | ✅ 3 題進階（diet_method / breakfast_habit / afternoon_craving） | ✅ | 拿代碼 + quiz 借力 |
| `/tools/protein` | 蛋白質 | 計算型 | ✅ 3 題進階（breakfast / lunch / dinner_type） | ✅ | **quiz 借力為主**（拿代碼 3.3% vs quiz 6.6%）|
| `/tools/waist-hip` | 腰臀比 | 計算型（腰圍 + 臀圍） | ❌ 無 | ❌ 無代碼流程 | LINE generic CTA（「之後開放體驗時優先通知」）|
| `/tools/blood-sugar` | 血糖穩定度 | 風險評估（症狀勾選） | ✅ 2 題進階（drink_habit / lunch_habit） | ✅ | 拿代碼路 |
| `/tools/fatty-liver` | 脂肪肝風險 | 風險評估（6 題 0-3 分） | 🟫 **2026-04-29 拿掉**（看結果直接 claim） | ✅ | 拿代碼路（最強 13.8%） |
| `/tools/insulin-check` | 胰島素阻抗 | 風險評估 | ❌ 無 | ❌ 無代碼流程 | LINE generic CTA |
| `/tools/sugar` | 每日糖攝取 | 計算型（體重 + 杯數 + 甜度） | ✅ 進階（drink_time / topping） | ✅ | 拿代碼路 |

### 設計差異原則

| 工具型態 | 該不該保留進階？ | 原因 |
|---|---|---|
| 計算型（蛋白質 / 糖） | ⚠️ 待測 | 計算型工具承諾感低，可能適合 quiz 借力 |
| 風險評估型（脂肪肝 / 血糖 / 胰島素） | ⚠️ 待測 | 已有風險警示鉤住用戶，承諾感高 → 拿代碼路效率好 |
| TDEE | ⚠️ 混合 | 計算型但量大，目前進階保留觀察 |

> 04-29 fatty-liver 拿掉進階是 v1 假說「風險評估型工具用 6 題 answers[] 直接展開個人化即可，進階兩題反而是阻力」的試驗。1-2 週後若數字提升，sugar / blood-sugar / protein 比照辦理（暫時不動 TDEE，量太大改動風險高）

### 3.4 已知不一致（Pattern 8 待清）

- **protein 命名歷史包袱**：`protein_calculate` 是「看結果」、`protein_complete` 是「拿代碼」 → 跟其他工具命名 swap。要改的話 GA4 探索報表要分段對照（觸發條件：下次大版本改 protein 漏斗時順便處理）
- **waist_hip 缺 complete 事件**：算出結果時沒 fire `waist_hip_complete`，無法看「多少人完成計算」（觸發條件：下次改 waist_hip 工具時補上）
- **waist_hip / insulin_check 用 generic `click_line_cta` + source**：**這個是有意設計**（CLAUDE.md line 66）。`click_line_cta + source` = generic fallback 給「沒 claim_code 流程」的工具用；`click_line_<tool>` (5 個) = 精準分流給「有 claim_code 領取流程」的工具用。**不應該收斂或反向拆**——兩種命名各有用途，混合是設計。改動會破壞既有 GA4 funnel 精準度

## 5. 個人化文案規則

### 5.1 文案資料來源

每個工具的個人化文案有兩層：

| 層 | 在哪 | 用什麼資料 |
|---|---|---|
| **網站端**（用戶看結果時看到） | abc-website 工具 page.tsx | 用 React state（answers / drinkHabit / 等）動態組合 |
| **LINE 端**（用戶領完代碼後收到） | official-yihugh-line-bot route.js | 從 Supabase `_sessions` 表撈 row → 字典 lookup |

### 5.2 兩端必須對應的字典

> ⚠️ **動手紅線**：改任一邊字典必須同步改另一邊，否則網站承諾跟 LINE 兌現會不對齊。

| 工具 | 網站字典 | LINE 端字典 | trigger 真相來源 |
|---|---|---|---|
| **fatty-liver** | `GAP_LINES`（page.tsx 約 line 163）| `FATTY_LIVER_GAP_LINES` + legacy `DRINK_AHA` / `LUNCH_AHA` + `FATTY_LIVER_RISK` | 6 題 `QUESTIONS.options.label`（label 字串就是 trigger）|
| **blood-sugar** | （網站端目前無對應 GAP_LINES，個人化文案散在 page.tsx if/else）| `BLOOD_SUGAR_DRINK_AHA` / `BLOOD_SUGAR_LUNCH_AHA` / `BLOOD_SUGAR_RISK_DATA` | `drink_habit` (4 enum) / `lunch_habit` (4 enum) / `risk_level` (4 enum: low/moderate/high/very-high) |
| **sugar** | （網站端目前無對應 GAP_LINES）| `SWEETNESS_GRAMS` (5 enum) / `CUPS_NUM` (4 enum) | `sweetness_level` / `cups_per_day` |

> **未來各工具改成「拿代碼直接領」pattern 時**（仿 fatty-liver），會把字典結構也對齊：網站建 `GAP_LINES` + LINE 建 `<TOOL>_GAP_LINES`，這份契約對應節要更新

## 6. Supabase Schema

### 6.2 共用欄位

- `id`：sessionId（前綴 + base36 random，例 `td_xx` / `fl_xx`）
- `claim_code`：4 字英數隨機（A-Z 排除 0/I/O/1，數字 0-9 排除 0/1，避免肉眼混淆），24 小時內有效
- 衝突重試：寫入 409 → 重試一次（換新 sessionId + 新 claim_code）
- `claimed_by` + `claimed_at`：**LINE 端領取後寫入**（網站端不寫）

### 6.2.1 NULL 邏輯規則（避免 Pattern 6 NULL 邏輯雷）

**`claimed_by` / `claimed_at`**：

| 狀態 | 含義 | LINE 端行為 |
|---|---|---|
| `claimed_by IS NULL` | 還沒被任何 LINE 帳號領過 | 允許領取 → UPDATE 寫入 LINE userId 跟 timestamp |
| `claimed_by = <某 LINE userId>` 且符合當前用戶 | 同人重複領（網路重連等）| 直接重發報告（不寫入第二次）|
| `claimed_by = <別的 LINE userId>` | 已被別人領過 | **目前無嚴格防衛**：理論上 4 字代碼洩漏可被人搶領。實作上 LINE 端是否禁止 = 待確認（建議：直接重發 + log，不報錯，因為代碼 24h 過期短）|

**legacy 欄位 fallback 行為**（針對 fatty-liver `drink_habit` / `lunch_habit`）：

- 04-29 後新代碼**不寫入**這兩欄，但歷史 row（04-29 前）**有值**
- LINE 端 `buildFattyLiverReport` 升級後（04-29 commit `a76ffae`）**完全不讀**這兩欄，**100% 改用 `answers[]` + `FATTY_LIVER_GAP_LINES`**
- 結果：**新舊 row 一視同仁走新邏輯**。歷史 row 的 `drink_habit / lunch_habit` 變成歷史紀錄欄位（純資料保留，無讀取者）
- 例外：如果未來想做「歷史用戶 vs 新用戶 LINE 訊息品質對照分析」，可手動 SELECT 比較 — 但**一般 LINE webhook 流程不讀這兩欄**

**WHERE 子句的 NULL 陷阱**（給未來 admin SQL）：
- `WHERE drink_habit = 'sugar-tea'` → NULL row 永遠不命中（PostgreSQL `NULL = anything` 永遠是 NULL）
- `WHERE drink_habit IS DISTINCT FROM 'unknown'` → NULL row 命中（IS DISTINCT FROM 把 NULL 當值處理）
- 寫統計查詢時要明確 `IS NULL` / `IS NOT NULL` 過濾，否則新版用戶（NULL）會被靜默排除

### 6.3 schema migration 規則

- 加新欄位 → 必須 NULLABLE（向下相容歷史 row）
- 拿掉欄位 → **不要 DROP COLUMN**，標 legacy 即可（避免歷史 row 撈起來爆 + 保留歷史分析能力）
- 欄位重新命名 → 不允許（會破壞 LINE 端 lookup）

### 11.1 數據觀察期（2026-05-01 ~ 05-12）

- baseline `fatty_liver_complete` → `click_line_fatty_liver` 全段轉換率 = 13.8%

### 11.2 工具一致化（觀察期後若數字 OK）

**觸發條件**：§11.1 觀察期結果 — fatty-liver 全段轉換率 ≥ 13.8% baseline 且絕對 LINE 點擊人數提升

按優先序處理（每個都要按 §10 鐵律附 baseline + 假說 + GO/NO-GO）：
1. **blood_sugar**（觸發：fatty-liver 觀察期通過）— 比照拿掉進階 + 升級 LINE 端用 `answers[]` 展開（要先補網站端 `BLOOD_SUGAR_GAP_LINES` 字典）
2. **sugar**（觸發：blood_sugar 觀察期通過）— 同上
3. **protein**（**不在優先序**）— 量大且 quiz 借力路本來就贏（6.6% vs 拿代碼 3.3%），若拿掉進階打亂 quiz 路會破壞既有最強路徑
4. **TDEE**（**不在優先序**）— 量最大、進階已是 NULLABLE 無強制，改動風險不對稱

### 11.3 字典盤點（§5 TBD 處）

### 11.5 schema 確認

✅ **已查證（2026-04-30 用 Supabase MCP）**。§6 已更新為實際 schema。

**查證時新發現**：
1. **5 個表都有 `claimed_by` + `claimed_at`** — LINE 端領取後寫入，網站端 payload 沒寫這兩欄（這是設計，網站只負責寫入時拿 claim_code）
2. **`blood_sugar_sessions.risk_level` 是 4 級**（low/moderate/high/very-high）— 比 fatty-liver 的 3 級多一個
3. **`sugar_sessions` 有 `lunch_habit` + `afternoon_snack` 欄位**但 page.tsx payload 沒寫入 — 可能是過去版本殘留欄位。**待釐清**：要清掉還是要補寫入？
4. **`protein_sessions` 沒有 `gender` 欄位**（跟 TDEE 不同）— 蛋白質計算用 `food_type` 而不是 gender 區分
5. **TDEE 進階三題（diet_method / breakfast_habit / afternoon_craving）都是 NULLABLE** — 表示 schema 設計上 TDEE 進階就是可選（用戶可不填進階直接拿代碼），跟 fatty-liver 拿掉進階前的「強制進階」設計不同。**這意味 TDEE 比 fatty-liver 早一步具備「拿掉進階」的能力**，未來重構成本最低
