## 4 致命洞（必修，按優先序）

### 🔴 洞 1（最致命）：§10 鐵律無檢核機制

**問題**：v1.1 §10 寫了「加進階 / 改漏斗階段前必須有 baseline + 假說 + GO/NO-GO 線」「commit message 必須附上」，但**無任何檢核機制**。歷史 commit `d92401b` 沒附就過關了；下次也會。

**修法**（**最高槓桿，30 分鐘成本**）：
1. 在 `abc-website/.husky/commit-msg` 或 `.git/hooks/commit-msg` 建立 hook
2. 邏輯：
   - 抓 staged files，判斷是否動到 `src/app/tools/*/page.tsx`（漏斗階段相關檔案）
   - 若是 → grep commit msg 是否含 `baseline` / `假說` / `GO` 三關鍵字（任一缺則 reject）
   - 若不是 → 放行
3. 加 `--no-verify` 跳過提示：「鐵律 §10 規定提供 baseline / 假說 / GO 才能繞過」
4. 部署：先在 abc-website 建，official-yihugh-line-bot 看一休要不要也建

**驗收**：故意 commit 一個碰 page.tsx 但沒附 baseline 的 commit，應被 reject。

### 🔴 洞 2：§11 觸發條件鏈死

**問題**：v1.1 §11 6 個 TODO 全部觸發條件「下次改該工具時順便處理」，若 fatty-liver 觀察期不通過 → blood_sugar / sugar 不會推進 → §11.3 字典盤點 / §11.4 命名一致化全部永久卡。比 v1「未來」還糟（v1 至少明說沒 deadline）。

**修法**（純文件，10 分鐘）：
- 每條 TODO 加**雙觸發**：
  - (a) 主觸發 — 現有「順便處理」式
  - (b) **死線觸發** — 無條件 6 週後 review，回答「做 / 不做 / 延後」三選一，記錄理由
- 死線寫進 `~/Library/LaunchAgents` 或 launchd reminder（看 BRAIN.md 「launchd排程總覽.md」可參考 system_git-old-backup清理.md pattern）
- 或最低限度：每條 TODO 標 `🗓 review-by 2026-06-11`（v1.1 寫日期 04-30 + 6 週）

**驗收**：6 週後（2026-06-11）有自動 reminder（系統提醒或文件中明確標出）。

### 🔴 洞 3：§5.2 跨 repo 字典同步無原子性

**問題**：v1.1 §5.2「兩處字典必須同步」是純規則。兩 repo 兩個 commit，中間任何一個 push 失敗 = 字典撕裂。Pattern 7 跨層原子性。

**修法（兩個選項擇一）**：

**選項 A**（推薦，30 分鐘）：abc-website pre-commit hook
1. `.husky/pre-commit` 偵測 `src/app/tools/fatty-liver/page.tsx` 的 `GAP_LINES` 是否變動
2. 若變動 → 強制提示「official-yihugh-line-bot 對應字典 commit hash 是？」
3. commit msg 必須含 hash（grep `official-bot:` 字樣 + 7 字 hash）→ 缺則 reject
4. 同邏輯放 official-bot 反向

**選項 B**（半天）：抽共用 npm package
- 把 `FATTY_LIVER_GAP_LINES` 抽到私有 npm package（例 `@yihugh/abc-content`）
- 兩 repo 都 import
- 改字典只改 package + bump 版本

**選擇建議**：先 A（hook，槓桿最大）。Package 等到第 3 個工具加進來時再做。

**驗收**：故意改 fatty-liver page.tsx 的 GAP_LINES 但 commit msg 沒附 official-bot hash → reject。
